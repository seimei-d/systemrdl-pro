## Problem Statement

Hardware-инженеры, пишущие SystemRDL для описания register maps, тратят значительное
время на ручной цикл проверки: правка `.rdl` файла → запуск `peakrdl html` в терминале →
переключение в браузер → reload страницы. Каждая итерация рвёт фокус, а ошибки
парсинга/elaboration видны только в терминале постфактум. Команда автора (несколько
конкретных коллег) явно просила инструмент, который убирает этот цикл и показывает
карту памяти live прямо рядом с кодом.

Сообщество SystemRDL живо (~4-5k stars на `systemrdl-compiler`), но полноценного LSP
+ интерактивного viewer'а в open source сегодня нет. Существующий `vscode-systemrdl`
от SystemRDL org даёт только TextMate grammar — это не тот инструмент, о котором речь.
Коммерческие альтернативы (Agnisys IDesignSpec, Semifore CSRCompiler) стоят десятки
тысяч $/год и неудобны.

## What Makes This Cool

- **Editor-agnostic LSP** — Vim/Neovim/Helix/Emacs-юзеры (а это половина hardware-мира)
  получают полноценный язык даром. VSCode-юзеры получают тот же LSP плюс встроенный
  webview с картой.
- **Live карта памяти** — правишь `.rdl`, через ~300мс видишь обновлённый layout
  с цветными зонами reset/sw/hw access, вычисленными адресами, развёрнутыми параметрами.
- **Source map в обе стороны** — клик на регистр в карте прыгает к строке исходника;
  hover на имени в редакторе подсвечивает ячейку в карте.
- **Foundation for more** — через тот же source map потом легко достроить live чтение
  с FPGA через JTAG/OpenOCD, или two-way editing (правка карты → правка кода).

## Constraints

- **Нет своего парсера.** `systemrdl-compiler` (Python, MIT) делает 80% работы:
  парсинг, elaboration, source locations, error reporting. Реинвенция = минус 1-2 месяца.
- **Live elaboration требует валидного top-level addrmap.** Пока пользователь правит,
  файл часто в невалидном состоянии. Стратегия: сохранять "last good elaboration"
  и показывать diagnostics поверх карты, а не обнулять её.
- **Python зависимость** — приемлемая (pyright, ruff, basedpyright — у всех есть Python),
  но требует bootstrap: автоматическая установка через `pip install systemrdl-lsp`
  при первом запуске extension.
- **Marketplace publish gate** — конкуренция с существующим `vscode-systemrdl`.
  Решение: другое имя (`systemrdl-pro` или `systemrdl-lsp`), явное упоминание разницы
  в README.

## Premises (все подтверждены пользователем)

1. **Реальный pain** — коллеги в команде автора напрямую просили инструмент. Не гипотеза.
2. **Open source — правильная форма** — сообщество SystemRDL достаточно велико для traction.
3. **systemrdl-compiler как backend** — не пишем свой парсер на TS/Rust.
4. **Editor-agnostic с самого начала** — LSP + standalone viewer, VSCode-обёртка как extension.

## Approaches Considered

### Approach A: Webview Wrapper (rejected — слишком слабый whoa)
Минимальный VSCode extension, который spawnит `peakrdl html` на save и эмбедит результат.
Effort S, но нет diagnostics в редакторе и VSCode-only. Не оправдывает open-source effort.

### Approach B: LSP + Standalone Viewer + VSCode Embed (CHOSEN)
Три компонента: Python LSP-сервер поверх systemrdl-compiler, standalone web viewer
(SPA + CLI `rdl-viewer file.rdl --serve`), и тонкий VSCode extension, который
запускает LSP-клиент и эмбедит viewer в webview через iframe. Effort L (4-6 недель с CC).

### Approach C: PeakRDL Studio — full web-IDE (rejected — over-scope)
Полный browser-based IDE с pyodide/WASM-сборкой systemrdl-compiler. Категориально
новый продукт, но XL effort и не решает изначальный pain быстро. Trap для перфекциониста.

## Recommended Approach

**Approach B**, с fast-ship стратегией: walking skeleton за неделю, расширение по неделям.

### Build sequence (fast-ship)

**Week 1 — Walking skeleton (publishable)**
- `systemrdl-lsp` v0.1: pygls + одна фича = `publishDiagnostics`
- VSCode extension: LSP-клиент + webview panel, рендерит результат `peakrdl html` (на save)
- Marketplace publish с явным README: "early version, LSP diagnostics + peakrdl-html viewer"
- **Outcome:** коллеги уже могут пользоваться. Diagnostics в редакторе + viewer без alt-tab.

**Week 2-3 — Полноценный LSP**
- `hover` (resolved address, width, sw/hw access)
- `documentSymbol` (outline в sidebar)
- `definition` (goto-def для component types, параметров)
- `incl_search_paths` через `systemrdl-lsp.includePaths` setting в workspace
- **Outcome:** Vim/Neovim/Helix-юзеры получают рабочий LSP

**Week 4-5 — Standalone live viewer**
- `rdl-viewer` Svelte SPA: дерево addrmap → regfile → reg → field, цветные зоны access
- Custom JSON-RPC `rdl/elaboratedTree` для push elaborated AST из LSP в viewer
- Watch-mode: viewer переподписывается на LSP при reconnect
- VSCode extension меняет webview iframe со статичного peakrdl html на http://localhost:5173

**Week 6 — Source map both ways**
- Клик на регистр в viewer → POST на LSP → VSCode reveals source range
- Hover на имени в редакторе → highlight cell в viewer (через kustom decoration)
- "Last good elaboration" на случай syntax error: viewer не пустеет, показывает stale карту + warning bar

## Open Questions

1. **TextMate grammar** — форкать существующий `SystemRDL/vscode-systemrdl` (fork +
   credit + improvements) или писать с нуля? Скорее форк.
2. **`incl_search_paths` UX** — workspace setting `systemrdl-lsp.includePaths`
   или auto-discovery из `peakrdl.toml` если он есть в repo? Скорее оба, fallback.
3. **Perl preprocessor** — игнорировать (как `systemrdl-compiler` сейчас) или
   добавлять через external `m4`/CPP-стиль pre-step? Игнорировать в MVP, документировать.
4. **Marketplace name** — `systemrdl-pro`, `systemrdl-lsp`, `peakrdl-vscode`? Решить до Week 1 publish.
5. **Лицензия** — MIT (совместимо с systemrdl-compiler) или Apache-2.0? Скорее MIT.

## Success Criteria

- **Week 1:** Marketplace publish, ≥1 коллега в команде установил и не вернулся к
  старому workflow.
- **Week 6:** Все 4 LSP-фичи работают в Vim/Neovim/Helix через mason/built-in клиент.
  Standalone `rdl-viewer` работает без VSCode. Source map двусторонний.
- **Quarter 1 после publish:** ≥50 installs на marketplace, ≥1 GitHub issue от
  внешнего пользователя (не из команды), ≥3 stars от Hardware-разработчиков
  (не от друзей).

## Distribution Plan

- **systemrdl-lsp** (Python пакет): PyPI publish через GitHub Actions on tag
- **rdl-viewer** (CLI + npm package): npm publish, brew tap (опционально)
- **vscode-systemrdl-pro** (extension): VS Code Marketplace publish через `vsce`
- **CI/CD:** GitHub Actions, monorepo с тремя пакетами или три отдельных репо
  (склоняюсь к monorepo для совместного version bump'а + единый changelog)
- **Установка для пользователя:** `code --install-extension <name>` → extension
  на старте проверяет наличие `systemrdl-lsp` в PATH и предлагает auto-install
  через `pip install systemrdl-lsp` (как pyright делает)

## Dependencies

- **systemrdl-compiler** (MIT, Python) — backend, обязателен
- **pygls** (Apache-2.0, Python) — LSP framework
- **VSCode LSP client API** — стандарт, без блокеров
- **PeakRDL-html** (используется только в Week 1 как временный рендерер) — отказываемся в Week 4

## The Assignment

**До конца этой недели: dogfood-эксперимент за 2 часа.**

Не пиши код. Сядь рядом с одним из коллег, который просил инструмент. Пусть он
поработает с .rdl файлом в своём обычном workflow 30 минут. Молча наблюдай.
Считай: сколько раз он переключался в терминал/браузер, сколько раз искал
адрес мысленно, какие ошибки заметил *после* commit а не *во время* написания,
какие кейсы с `include` он использует.

Принеси этот список наблюдений как input для Week 1 scope. Это будет лучшим
predictor того, шиппить тебе walking skeleton именно с `publishDiagnostics`,
или людям вообще-то важнее `documentSymbol` или `goto-def`.

В hardware-разработке "watch don't demo" работает x2 — у инженеров мало
терпения объяснять, что им нужно, но много терпения молча мучиться.

## Viewer UX (from /plan-design-review on 2026-04-29)

### Approved direction — Variant B: Tree + Detail Pane

VSCode-native split. Left: collapsible tree (addrmap → regfile → reg → field).
Right: rich detail pane for selected register (all fields + descriptions + source link).
Mockup at `~/.gstack/projects/systemrdl-vscode/designs/viewer-panel-20260429/variant-B-tree-detail.html`.
This becomes the source of truth for Week 4-5 Svelte components.

Two alternatives explored and rejected: Variant A (dense spreadsheet — best density but
hover-highlight ambiguity, field detail needs popover) and Variant C (visual address bands —
best for spatial gaps but variable-height virtualization adds 1-2 weeks to Week 4-5).

### Design Decisions (D4-D15)

| ID | Decision | Notes |
|----|----------|-------|
| D4 | Auto-select first register on tab open | No empty pane on first visit |
| D5 | Tab overflow: «…» menu + Cmd-P fuzzy filter | Scales to 100+ addrmap roots |
| D6 | Tree default: regfile expanded, regs collapsed | Auto-expand the regfile of selected reg |
| D7 | Stale state: keep last-good map + amber bar `⚠ … [Show error] [×]` | Zero flicker on syntax errors |
| D8 | Library-only state: warm empty state with explanation + docs link | Krug "empty states are features" |
| D9 | Partial state: amber dot on broken tab + tooltip `parse error in <file>:<line>` | VSCode-native problem-dot pattern |
| D10 | Cursor sync (editor → viewer) with 500ms debounce | Vim users get magic without mouse |
| D11 | Typography: Inter (chrome) + JetBrains Mono (data) bundled as WOFF2 | Locks visual identity, hex digits aligned |
| D12 | Access mode palette locked (Mockup B values, see Design Tokens below) | Hardware-conventional muted palette |
| D13 | Auto-stack at < 700px viewport (tree on top, detail below) | Right-dock side-by-code use case works |
| D14 | Light-theme tokens locked in parallel | No CSS re-architect in Week 5 |
| D15 | Cmd-F inline filter input over tree | Scales to 200+ regs per tab |

### Interaction State Coverage (Pass 2)

| State | UX |
|-------|-----|
| Loading (LSP boot) | Subtle spinner + "Starting LSP…" in status bar; tree empty placeholder |
| Empty workspace (0 .rdl) | Command no-op (test plan edge case) |
| Library-only (.rdl без addrmap) | Warm empty state: explanation + `Open docs: include & elaboration` link (D8) |
| Parse error, no last-good | Hard error in detail pane: error message + line + `[Open file]` action |
| Parse error, last-good exists | Last-good map kept, amber stale-bar above tabs (D7) |
| Success | Mockup B normal state |
| Partial (1 of N addrmap failed) | Amber dot on broken tab + tooltip; click shows last-good for that tab (D9) |
| LSP crashed | Banner across panel: `LSP server stopped. [Restart LSP]` (eng review #1) |
| Python module missing | Banner: `systemrdl-lsp not installed. [Install] [Choose Python…]` (2B chain) |
| Python interpreter wrong | Banner: `Python 3.10+ not found. [Set systemrdl-pro.pythonPath…]` (2B) |
| Webview disposed mid-push | Silent — `if (panel.visible) postMessage(…)` guard (eng review #2) |

### User Journey (Pass 3) — 5-sec / 5-min / 5-year

- **5 sec** — install → open .rdl → diagnostics appear → click "Show Memory Map" →
  panel opens with tree expanded one level + first register auto-selected in detail pane.
  Visceral: "this works".
- **5 min** — edit register width, see live update, click field in viewer, editor jumps
  with smooth scroll + line flash. Type Cmd-F, filter to "DMA", arrow to register, see fields.
  Behavioral: flow.
- **5-year** — second nature, keyboard-driven. Cmd-P swaps roots. Auto-follow cursor (D10)
  keeps viewer in mirror with editor. Reflective: invisible tool.

### Responsive (Pass 6)

- ≥ 700px wide: side-by-side tree (resizable, default 320px, min 240px) + detail pane.
- < 700px wide: auto-stack — tree on top (default 50% height, resizable horizontal divider),
  detail below.
- Tab strip: overflow menu kicks in when total tab width > viewport (D5).

### Accessibility (Pass 6)

- **Keyboard**: Tab cycles 4 regions (tab strip → filter → tree → detail). Tree: Up/Down nav,
  Right expand, Left collapse, Enter select. Tabs: Arrow ←/→ between, Cmd-P open overflow filter.
  Esc dismisses stale-bar / clears filter.
- **ARIA**: `role="tablist"` on tab strip, `role="tab"` per tab. `role="tree"` + `treeitem`
  with `aria-expanded` / `aria-selected`. Stale-bar `role="status" aria-live="polite"`.
  LSP/Python banners `role="alert"`.
- **Color-blind**: redundant text encoding for every color signal. Access labels ("RW", "RO")
  always visible in tree + detail. Selection has color + 3px left border. Stale-bar has
  color + ⚠ icon + text.
- **Contrast**: all fg/bg pairs verified ≥ 4.5:1 in mockup. Light theme verified at
  implementation time. High-contrast tokens deferred (TODO-D2).

### Locked behaviors from Pass 7 (U2/U4/U5)

- **Source-link click** (U2): smooth `revealRange` + 200ms line-flash decoration on target.
- **No bit-strip cell hover popover** (U4): cells are 6px wide; redundant with detail pane
  visible for the selected register. Accept loss.
- **Right-click context menu** (U5): native VSCode `vscode.window.showQuickPick` with
  Copy Address / Copy Name / Reveal in Editor.
